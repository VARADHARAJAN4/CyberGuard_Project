# CyberGuard AI Agent — Complete Working Principle (How It Works)

This project implements a **Cybersecurity Monitoring AI Agent** called **CyberGuard**. It continuously processes security events, detects suspicious behavior, assigns risk to incidents, generates recommendations, and serves everything through a **Streamlit dashboard**.

---

## 1) Data Flow (End-to-End Pipeline)

### A. Log Ingestion (Data Collection)
- The project uses **sample/synthetic security events** (a stand-in for real firewall / endpoint / email / AD / syslog events).
- These events are generated by `cyberguard/data_generator.py`.

**Event fields stored per record include**:
- `timestamp`
- `source` (e.g., firewall, windows_event)
- `event_type` (failed_login, malware_detected, port_scan, etc.)
- `severity` (low/medium/high/critical)
- `source_ip`, `dest_ip`
- `username`, `department`
- `protocol`, `port`
- `status` (normal or attack)
- `details`, `raw_log`

### B. Storage (Database Layer)
- Events and derived alerts are stored in **SQLite**.
- The DB schema is created in `cyberguard/database.py` inside `init_db()`.

Tables:
- `security_events`: raw normalized security events
- `alerts`: prioritized alerts derived from events
- `incident_reports`: generated incident reports

### C. ML / Analytics (Detection & Scoring)
CyberGuard performs analysis using two layers:
1. **Heuristic risk scoring** (always available)
2. **Unsupervised anomaly detection** (optional until trained)

#### 1) Heuristic Risk Score
- Implemented in `cyberguard/anomaly_detector.py` as:
  - `compute_risk_score(event)`
  - `classify_risk_level(score)`

How it works:
- Base score comes from `severity`
- Score increases for known attack patterns in `event_type`
- Score boosts when `status == "attack"`
- Score adds extra weight if `port` is suspicious (SSH/RDP/DB ports etc.)

Output:
- `risk_score` in range **0.0 to 1.0**
- `risk_level` (low/medium/high/critical)

#### 2) Isolation Forest + Local Outlier Factor (Optional)
- Implemented in `cyberguard/anomaly_detector.py` as the `AnomalyDetector` class.
- Training is performed by calling `CyberGuardAgent.fit_detector()`.

Models:
- **IsolationForest** for anomaly isolation
- **LocalOutlierFactor** (novelty=True) for local anomaly detection

Feature engineering:
- Converts categorical values (source, event_type, severity, etc.) to numeric encodings
- Adds time-derived features: `hour`, `day_of_week`, `is_weekend`, `is_business_hours`
- Adds port-based indicators: `is_suspicious_port`

When ML is not trained, the system still works using the heuristic scoring.

### D. Reasoning & Recommendations (Agent Layer)
- Implemented in `cyberguard/agent.py` as `CyberGuardAgent`.

The agent produces:
- **Plain-language explanation** for an event (`analyze_event`)
- **Attack prediction** (`predict_attack`) using ML anomalies + risk score
- **Remediation actions** (`recommend_actions`) using templates per alert type
- **Incident report generation** (`generate_incident_report`)

### E. Presentation (Streamlit Dashboard)
- Implemented in `cyberguard/dashboard.py`.

Dashboard pages:
- **Overview**: KPIs, charts, timeline, top attackers
- **Event Analysis**: training button + analysis button + single-event explanation
- **Alerts**: filter alerts and show recommended actions
- **AI Agent Chat**: query-based answers (rule-based summaries)
- **Incident Reports**: previously generated incident reports
- **Settings**: seed sample data + reset DB + train ML

---

## 2) How Alerts Are Created (From Events)

During seeding (`seed_database()` in `cyberguard/data_generator.py`):
1. The generator creates events (normal vs attack) with realistic patterns.
2. If an event is an attack (`status == "attack"`), the generator:
   - Computes `risk = compute_risk_score(event)`
   - Assigns `risk_level` thresholds (critical/high/medium)
   - Stores an `alerts` record linked to the event’s `event_id`

So the dashboard doesn’t wait for ML training to show alerts—alerts are produced immediately from heuristic risk.

---

## 3) Threat Prioritization (Risk Levels)

Risk levels used across the system:
- **critical**: score ≥ 0.8
- **high**: score ≥ 0.6
- **medium**: score ≥ 0.3
- **low**: otherwise

The Streamlit UI sorts alerts by `risk_score desc` and highlights higher severity.

---

## 4) What the Agent Says (Example Reasoning)

The agent’s outputs are intentionally SOC-style:
- “Possible brute-force attack in progress”
- “Malware alerts on X workstations”
- “Data exfiltration attempts detected. Immediate investigation required”

Recommendations are action-oriented, mapped per alert type, e.g.:
- failed_login → block IP, enable MFA, force password reset
- port_scan → block scanning IP, run vulnerability scan
- malware_detected → isolate host, run AV/EDR scan
- data_exfiltration → block outbound, revoke credentials, start incident response

---

## 5) Typical Usage Workflow (End User)

1. Launch the system
   - Seed sample data from Settings (or run `python run.py --seed-only`).
2. Open dashboard
   - `python run.py --dashboard`
3. Use it in three ways:
   - Browse KPIs & timeline
   - Inspect alerts and recommended actions
   - Train ML (optional) and run deeper event analysis
4. Generate an incident report
   - Use AI chat query or the report generator flow

---

## 6) Why This Works (Design Intent)

- **Operational robustness**: even without ML training, heuristic scoring still produces alerts and explanations.
- **Explainability**: recommendations are deterministic templates; outputs are easy to interpret.
- **Extensibility**: replace synthetic generator with real log ingestion while keeping the same pipeline.

---

## Repository Entrypoints

- `cyberguard/agent.py`: analysis, recommendations, incident report generation
